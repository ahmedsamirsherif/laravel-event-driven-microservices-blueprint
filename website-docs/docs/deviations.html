Key Decisions in Detail

ADR-003: Pragmatic DDD

Eloquent models are permitted in the Domain layer. Full DDD (separate repository + domain model objects) was considered too heavy for a two-service project. This keeps the codebase lean while maintaining layered architecture benefits — the Domain layer still owns the interface contracts, value objects, and domain events.

ADR-006: Auto-Discovering Country Strategy

Convention-based CountryClassResolver replaces manual registration. It scans Domain/Country/*/ directories, builds the FQCN as App\Domain\Country\{ISO3}\{ISO3}{Suffix}, and validates the interface via reflection. Adding a country = adding a single directory with one file. No service provider changes, no registry updates required.

ADR-009: Dead Letter Queue

Failed messages route to a DLQ after 3 retry attempts with exponential backoff (1 s → 2 s → 4 s). This prevents poison messages from blocking the main queue while preserving them for manual inspection and replay. The DLX is a standard fanout exchange — any tooling can bind to it for alerting.

ADR-011: Event Schema Contract

JSON Schema 2020-12 at contracts/employee-event.schema.json serves as the shared integration contract validated by both services in their contract test suites. The CountryCode enum is consciously duplicated in each service to avoid shared-library coupling — the schema file is the single source of truth, not a shared PHP package.

Beyond the Challenge Spec

The following features extend the base challenge requirements as deliberate enhancements, each implemented without modifying existing country or framework code.

Germany Documentation Tab (Extra Navigation Step)

The challenge required country-specific validation fields (Tax ID, Goal) for Germany. This implementation adds a third navigation step, Documentation, that only appears for DEU. It renders a document-tracking table against five German compliance document types — Work Permit, Tax Card, Health Insurance, Social Security, Employment Contract — with per-employee Yes/Missing status and an overall docs-completion percentage. The step is entirely server-driven: DEUModule::navigationSteps() includes the documentation entry; the blade SPA renders the tab automatically without any country conditionals in the frontend code.

Document Completion List for German Employees

loadDocumentationView() fetches employees and checklist data in parallel and renders a live matrix table. The percentage per employee is computed from checklist doc_* item state or directly from projected employee fields. No backend changes were required — the five doc_* fields are projected as part of the standard employee_projections row and the ChecklistService treats them as documentation checklist items.

Named Employee Rows with Per-Field Modal Breakdown

The challenge required a checklist endpoint showing overall completion. This implementation surfaces employee names throughout the Hub UI (checklist summary rows, checklist cards, documentation table) and makes every row clickable. The resulting modal breaks down completion by checklist step, then by individual field, showing ✅ / ⚠️ icons, the validation message for missing fields, and a field-key badge. This required extending ChecklistService::assembleChecklist() to include name and last_name.

In-Browser "Simulate Event" Button

A ▶ Simulate Event button in the Hub UI topbar POSTs a randomly generated employee (cycling through four personas with random country-specific fields) to the HR Service API. This triggers the full pipeline — HR validates → RabbitMQ publishes → Hub consumer processes → Redis cache invalidates → WebSocket broadcasts — demonstrating end-to-end flow without opening a separate terminal.
